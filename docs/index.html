Security

Defence in depth

Scriptomatic applies multiple independent security layers so that a failure in any one layer does not expose the site.

üõ°Ô∏è

Capability Gates

Every save callback verifies manage_options before processing any input. No data is written without this check passing.

üîë

Dual Nonce Verification

Both the WordPress Settings API nonce and a secondary per-location nonce must validate. One expired or missing nonce aborts the save.

‚è±Ô∏è

Rate Limiting

A transient-based 10-second per-user, per-location cooldown prevents rapid repeated saves or brute-force form submissions.

‚úÖ

Input Validation

Inline scripts capped at 100 KB (plugin-enforced). JS file size is limited by the server’s upload settings, not the plugin. Invalid UTF-8 and control characters rejected. <script> tags stripped. Dangerous HTML tags flagged with admin notice.

üîó

URL Validation

External script URLs are validated with wp_http_validate_url() and deduplicated before storage and before injection.

üìã

Activity Logging

All saves, rollbacks, file_save, file_delete, and restore events are logged with timestamp and user identity. Inline script and external URL changes produce separate independent entries, each restorable on its own. Configurable retention limit (3–1000). Oldest entries are discarded automatically once the cap is reached.

‚Ü©Ô∏è

Open Redirect Hardening

Post-action redirects use a constructed admin_url() rather than the HTTP Referer header, which can be spoofed by an attacker.

üß±

Singleton Guards

__clone() and __wakeup() methods prevent the singleton from being duplicated via object cloning or PHP unserialize().

üîí

Directory Index Files

Both the plugin root and includes/ directory contain an index.php that returns HTTP 403, preventing directory listing if web server options are misconfigured.

üåê

REST API Security

All 13 REST endpoints are POST-only. Authentication exclusively via WordPress Application Passwords (Authorization: Basic header ‚Äî credentials never in the URL). An optional IP Allowlist in Preferences blocks requests from outside a set of IPv4/IPv6 addresses or CIDR ranges before the capability gate is even reached (403 rest_ip_forbidden). Rate limiting and activity logging apply equally to REST and admin UI operations through the shared service layer.

REST API

Automate everything over HTTP Pro

All 13 endpoints are POST-only. Authentication uses WordPress Application Passwords ‚Äî credentials always travel in the Authorization: Basic header, never in the URL.

Authentication

Generate an Application Password under Users ‚Üí Profile ‚Üí Application Passwords. Then pass Authorization: Basic base64(username:app-password) with every request.

WP-CLI

Manage scripts from the command line Pro

The wp scriptomatic command group mirrors the full REST API. All write commands share the same service layer ‚Äî identical validation, rate-limiting, and activity logging apply.

Pro Feature

WP-CLI commands require an active Scriptomatic Pro licence. View pricing ‚Üí

Inline Script

$ wp scriptomatic script get --location=head $ wp scriptomatic script set --location=head --file=script.js $ wp scriptomatic script rollback --location=head --id=42 $ wp scriptomatic history --location=head --format=table

External URLs

$ wp scriptomatic urls get --location=footer $ wp scriptomatic urls set --location=footer --urls='[{"url":"https://cdn.example.com/lib.js","conditions":{"logic":"and","rules":[]}}]' $ wp scriptomatic urls rollback --location=footer --id=37 $ wp scriptomatic urls history --location=footer

Managed JS Files

$ wp scriptomatic files list $ wp scriptomatic files get --id=my-tracker $ wp scriptomatic files set --label="My Tracker" --file=tracker.js $ wp scriptomatic files upload --path=tracker.js --label="Tracker" $ wp scriptomatic files delete --id=my-tracker --yes

--format defaults to table; accepts table, json, csv, yaml, count.   Use the history commands to look up --id values for rollback.   --conditions accepts a JSON {logic, rules} object.

Release History

Changelog

Selected highlights ‚Äî see the full CHANGELOG.md on GitHub.

v3.0.0

Freemius integration ‚Äî freemium model, Scriptomatic Free + Pro

Scriptomatic is now freemium. Core WordPress JavaScript injection features (inline scripts, external URLs, activity log with rollback) remain free and unlimited. Conditional loading, Managed JS Files, REST API, WP-CLI, and the API IP Allowlist now require a Pro licence. A 3-day free trial (credit card or PayPal required) is available. The Freemius SDK (freemius/) is bundled for licence management, in-dashboard upgrade, and uninstall feedback. Uninstall cleanup is now hooked to the Freemius after_uninstall action rather than a static uninstall.php file. Free users see contextual upgrade notices on locked features. Bug fix: $conditions variable shadow in inject_scripts_for() corrected.

v2.9.0

Custom DB log table; rollback by row ID

The activity log is now stored in a dedicated {prefix}scriptomatic_log database table with indexed columns for fast, filtered reads — it no longer occupies a WordPress options row. All rollback paths (admin AJAX, REST API, WP-CLI) now identify a snapshot by its DB row id integer rather than a 1-based array position. Obtain IDs from the history endpoints before calling rollback. The table is dropped cleanly on plugin deletion.

v2.8.0

Unified location options

The six per-location wp_options rows (separate keys for script content, URL list, and conditions for each of head and footer) have been consolidated into two unified options — scriptomatic_head and scriptomatic_footer. Each stores a single PHP array containing script, conditions, and urls. Existing data is automatically migrated on first admin load; no manual action is required.

v2.7.0

JS Files list-page upload, REST API IP allowlist, and upload CLI/API endpoints

The JS Files list page gains a dedicated Upload a JS File card with a file picker and an Upload & Edit button. Submitting the form performs a real server-side upload and redirects to the edit page for review — no browser FileReader involved. A new API Allowed IPs field in Preferences lets you restrict REST API access to specific IPv4 addresses, IPv6 addresses, or IPv4 CIDR ranges — leave it empty to allow all IPs (the default). Two new upload entry points round out the release: POST /wp-json/scriptomatic/v1/files/upload accepts a multipart/form-data request, and wp scriptomatic files upload --path=<file> reads a local file and saves it through the shared service layer.

v2.6.0

REST API & WP-CLI command group

Every Scriptomatic operation is now available over a first-party REST API (scriptomatic/v1) and through a wp scriptomatic WP-CLI command group. All REST endpoints are POST-only — credentials always travel in the Authorization: Basic header via WordPress Application Passwords, never in the URL. The two interfaces share a public service layer (service_* methods) so validation, rate-limiting, and activity logging are identical regardless of how a change is made. Twelve REST routes cover inline scripts, external URL lists, and managed JS files, including read, write, rollback, and history. Twelve CLI commands mirror the same operations with table, JSON, CSV, and YAML output formats.

v2.5.3

External URL Restore polish; docblock audit; language consistency

The Restore button on external-URL log entries is now also disabled when the snapshot contains no URLs (e.g. a “URL removed” entry that cleared the entire list) — restoring it would simply wipe the current list for no benefit. User-facing “URL list” language replaced with “external URLs” throughout all tooltips, log detail strings, and success messages. Stale internal docblocks corrected: write_activity_entry() action list now includes url_save, url_rollback, and file_restored; ajax_rollback() no longer references the removed v2.5.0 combined-restore behaviour.

v2.5.2

Independent inline-script and external-URL Activity Log entries

Inline script + load-conditions changes and external URL changes are now written as fully independent Activity Log entries with separate index counters, View buttons, and Restore buttons — restoring one dataset never touches the other. The Restore button on the most recent entry of each dataset is disabled with a tooltip (“This is the current version — nothing to restore”). Conditions-only saves always produce a complete, restorable entry: when only conditions changed, the entry writer snapshots the current script from the database so every inline entry carries a full content + conditions snapshot.

v2.5.1

Ellipsis display fix; comprehensive code audit

Restore, Load, and Delete buttons were rendering the literal string Restoring\u2026 instead of Restoring…. Fixed by using the actual UTF-8 ellipsis character in the wp_localize_script i18n payload. Full audit of all PHP traits and admin JS; six additional issues corrected (escaping, nonce handling, rate-limit logic, AJAX error paths).

v2.5.0

Combined snapshot save; single-click full restore

Each Save now writes one combined Activity Log entry containing the inline script, external URL list, and load conditions — a complete snapshot of the location‚Äôs state. Clicking Restore on any entry brings back all three fields simultaneously via direct $wpdb writes, with no further Save needed. The separate url_added, url_removed, conditions_save, url_list_restored, and conditions_restored event types and their individual AJAX restore endpoints have been removed. The Changes column now shows a human-readable summary (Script: 245 chars · 1 URL added · Conditions: Front page only).

v2.4.0

File conditions UI fixed; duplicate conditions log entries fixed

The condition type picker on the Add/Edit File page now correctly renders the stacked-rule UI. render_file_conditions_widget() was still using the legacy single-select UI; it now delegates to render_conditions_stack_ui() with automatic migration of legacy {type,values} data. Also fixed a missing Settings API double-call guard in sanitize_conditions_for() that caused two identical conditions_save Activity Log entries to be written on every inline-script conditions save.

v2.3.0

Single-concern Activity Log entries; greyed-out Restore for removals

Each Activity Log action now records only its own data: url_added / url_removed log the URL list only; conditions_save logs conditions only; save / rollback log script content only. No more silent cross-restoration — restoring a URL list no longer also overwrites conditions, and vice versa. The Restore button is now disabled (with an explanatory tooltip) on url_removed, empty-content saves, and file_delete entries; View still works for audit reference. To undo a removal, find an earlier log entry and click Restore there.

v2.2.0

Inline-script conditions UI fixed; activity-log label clarity

The condition type selector on the Head Scripts and Footer Scripts pages now correctly renders sub-panels when a rule type is chosen. The underlying render_conditions_field_for() was rewritten to use the shared stacked-UI renderer, with automatic migration of any legacy {type,values} saved data. Activity Log entries now label inline-script condition snapshots as Inline Script Load Conditions and file conditions as File Load Conditions, eliminating the previously contradictory Load Conditions heading.
